## 😤 The Problem

After a bad deploy you're doing this:

1. Open GitHub → read the diff
2. Open Grafana → check if metrics spiked
3. Open CloudWatch / Loki → grep for errors
4. Open Notion / Confluence → write the postmortem from scratch

**You're stitching together 4 tools at 2am.** postmortem-pilot collapses all of that into one command.

---

## ✨ What It Does

```
$ postmortem-pilot run --log-file ./app.log --before HEAD~1 --after HEAD
```

1. 📂 **Reads the git diff** between two commits
2. 🔍 **Scans your logs** for errors, warnings, and anomalies
3. 🤖 **Calls Groq/Llama3** (free) to write a plain-English summary
4. 📄 **Writes `postmortem.md`** to your repo — versioned, searchable, permanent

In GitHub Actions, this runs **automatically after every push to main**.

## 🚀 Quick Start

### 1. Install

```bash
# pip install postmortem-pilot : roadmap, will be released soon
# or for development:
git clone https://github.com/eya-lachhab/postmortem-pilot
cd postmortem-pilot
pip install -e ".[dev]"
```

### 2. Get a free Groq API key

Sign up at [console.groq.com](https://console.groq.com) — no credit card required.

```bash
export GROQ_API_KEY=your_key_here
```

### 3. Run it

```bash
# Against your last deploy
postmortem-pilot run \
  --log-file ./app.log \
  --before HEAD~1 \
  --after HEAD

# With the bundled sample log (no log file needed)
postmortem-pilot run

# Template mode only (no API key)
postmortem-pilot run --no-llm

# Check your environment
postmortem-pilot doctor
```

## 🤖 GitHub Actions Setup

### Step 1: Add your Groq API key as a GitHub Secret

Go to your repo → **Settings → Secrets and variables → Actions → New repository secret**

- Name: `GROQ_API_KEY`
- Value: your Groq API key

### Step 2: The workflows are already in `.github/workflows/`

| Workflow | Trigger | What it does |
|---|---|---|
| `ci.yml` | Every push & PR | Lint (ruff) + type check (mypy) + tests (pytest) |
| `postmortem.yml` | Every push to `main` | Runs postmortem-pilot, commits `postmortem.md` back |

### Step 3: Push to main — that's it

After every deploy to main, a fresh `postmortem.md` will be committed to your repo automatically.

## 📄 Example Output

```markdown
# 🚨 Postmortem Report
> Generated by postmortem-pilot on 2026-05-10 14:07 UTC

## Deploy Summary
| Field       | Value      |
|-------------|------------|
| Files Changed | 3        |
| Insertions  | +42        |
| Errors      | 5          |

## AI Summary

**TL;DR:** Deploy introduced a DB connection pool exhaustion bug that caused
payment failures for ~3 minutes before auto-recovery via replica failover.

**What Changed:** Updated order handler with new retry logic; modified DB
connection pool settings; bumped requests library.

**Issues Detected:** 3x database connection timeout, circuit breaker opened,
payment-service connection refused (4x).

**Likely Root Cause:** New connection pool size setting reduced available
connections under load.

**Recommended Actions:**
1. Revert connection pool config or increase pool size
2. Add DB connection pool metrics to your dashboard
3. Test payment-service failover in staging before next deploy
```
